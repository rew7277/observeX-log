  "environment": "SANDBOX",
  "application": "demo-checkout-api",

{
  "status": "ok",

401 Missing token / Invalid API key
400 logs field required
413 Uploaded log is too large
500 Internal server error
Async Queue Ingestion
POST /api/v1/logs/ingest-async

Check jobs inside Data Source Health or GET /ingestion/jobs.
cURL
curl -X POST https://your-domain/api/v1/logs/ingest \
  -H "Authorization: Bearer obsx_live_xxxxxxxxxxxxxxxxxxxxxxxx" \
  -H "Content-Type: application/json" \
  -d '{"environment":"SANDBOX","application":"demo-checkout-api","logs":"INFO 2026-05-12 09:18:44 ..."}'
Success / Failure / Limits
Success: 200 { status, session_id, stored, health_score, log_rows }
Failure: 400 logs field required · 401 invalid API key · 413 max upload exceeded
Railway volume: set OBSERVEX_DATA_DIR=/data and mount a volume at /data.

Premium Custom Masking Rules
Add extra keys that should be masked before logs are displayed. This is a premium control for customer-specific sensitive fields like cardToken, employeeId, memberId, deviceId, sessionSecret, customerDOB, etc.
Custom masking is loaded from this browser. Backend persistence can be connected later for Enterprise tenants.


Security & Masking
ObserveX masks JWTs, bearer tokens, API keys, Aadhaar, PAN, mobile numbers, emails, customer names, loan/account/customer/checkout IDs, BBPS IDs, UPI/VPA and common secrets before rendering logs, exporting CSV, storing volume copies, or returning API responses.
For production SaaS: store only masked logs by default; allow encrypted raw retention only as an enterprise admin setting.
