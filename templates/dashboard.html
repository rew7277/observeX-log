⚡ API Ingestion
Send logs from Mule, CI/CD, cron jobs, S3 processors, or any backend service.
Authentication
Use your API key as a Bearer token.
Authorization: Bearer {{ user.api_key }}
POST /api/v1/logs/ingest
{
  "environment": "PROD",
  "application": "s-paymentengine-api",
  "logs": "INFO 2026-04-25 14:51:17 ..."
}
Success Response
{
  "status": "ok",
  "session_id": 123,
  "total": 10674,
  "errors": 0,
  "warns": 0,
  "apps": ["s-paymentengine-api"],
  "health_score": 100
}
Failure Responses
401 Missing token / Invalid API key
400 logs field required
413 Uploaded log is too large
500 Internal server error
cURL
curl -X POST https://your-domain/api/v1/logs/ingest \
  -H "Authorization: Bearer {{ user.api_key }}" \
  -H "Content-Type: application/json" \
  -d '{"environment":"PROD","application":"s-paymentengine-api","logs":"INFO 2026-04-25 14:51:17 ..."}'
Success / Failure / Limits
Success: 200 { status, session_id, stored, health_score, log_rows }
Failure: 400 logs field required · 401 invalid API key · 413 max upload exceeded
Railway volume: set OBSERVEX_DATA_DIR=/data and mount a volume at /data.

Environments
Security & Masking
ObserveX masks JWTs, bearer tokens, API keys, Aadhaar, PAN, mobile numbers, emails, customer names, loan/account/customer/payment IDs, BBPS IDs, UPI/VPA and common secrets before rendering logs, exporting CSV, storing volume copies, or returning API responses.
For production SaaS: store only masked logs by default; allow encrypted raw retention only as an enterprise admin setting.
